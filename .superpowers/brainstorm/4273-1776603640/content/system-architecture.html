Decomposition Question:

The game has natural layers: core logic (modules 1–6) and UI (Phase 1 then Phase 2). We also need persistence + mobile. How should we sequence this?

A

Core First (TDD), Then UI Layers

1. Core modules (all 6) with unit tests
2. GameEngine + integration tests
3. Phase 1 UI (single node)
4. Phase 2 UI (multi-node)
5. Persistence + Capacitor

Pros: Game logic is bulletproof before UI. Can iterate UI without touching core.
Cons: Long time before first playable moment.

B

Vertical Slice to Playable, Then Scale Up

1. NodeManager + ContractSystem (minimal)
2. SentimentSystem + Economy
3. GameEngine with tick loop
4. Phase 1 UI (playable MVP)
5. UpgradeSystem + shop UI
6. Phase 2 multi-node + UI
7. Persistence

Pros: Play the game early, iterate based on feel.
Cons: May refactor core as Phase 2 reveals issues.

C

Phase 1 Complete, Then Phase 2 (Waterfall)

1. Phase 1 Core (NodeManager, ContractSystem, Sentiment, Economy)
2. Phase 1 UI (single node, manual cool, basic upgrades)
3. Persistence for Phase 1
4. Phase 2 expansion (multi-node logic)
5. Phase 2 UI (node selector, power allocator)
6. Full persistence + Capacitor

Pros: Ship a complete game after step 3, add depth incrementally.
Cons: Rework potential if Phase 1 and Phase 2 designs conflict.
